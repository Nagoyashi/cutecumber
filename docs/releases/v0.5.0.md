# v0.5.0 — Public profile + auth refresh

The second half of the design refresh. v0.4.0 restyled the landing and the
dashboard; this cycle finishes the job on the two surfaces that were still
plain — the **public profile page** and the **auth screens** (login, signup,
password reset) — and adds the small state screens that make claiming a username
feel friendly.

Same hard rules as ever: zero JavaScript and zero third-party requests on public
pages, no cookies for anonymous visitors, every pill clears WCAG-AA contrast.

## Public profile

- **Centered + wide layouts.** The public page (`cutecumber.cc/you`) was
  restyled with a proper centered layout and an optional wide variant, plus a
  matching settings surface in the editor. (#50)
- **Page settings + circle-crop avatars.** New per-page theme controls — layout,
  ambient background, and the "made with cutecumber" credit toggle — landed as a
  versioned **theme v3** shape with a migration. Uploaded photo avatars are now
  circle-cropped, so they're visually distinct from the square drawn-set tiles.
  (#55)
- **Live avatar preview + credit text.** The editor previews a picked avatar
  immediately, and the credit line copy was tidied. (#62)

## Auth screens

- **A shared landing-shell.** Login, signup, and password-reset were recreated
  on one shared shell — centered slice wordmark + a glassy card on the ambient
  background — factored into a reusable Jinja partial so the dashboard and the
  auth pages share the same chrome. Auth pages stay `noindex`. (#51)
- **Show/hide password + a live username hint.** Both are progressive
  enhancements: every form still submits with JavaScript switched off. The
  username hint is a regex-only typo nudge — the server stays the sole authority.
  (#51)
- **Username-taken state.** A claim collision now re-renders the form in place
  with the attempted name kept, an error on the field, and a few **suggestion
  chips** — each one re-checked against live usernames and resting tombstones, so
  every suggestion shown can actually be claimed. The chips are plain links (zero
  JS), so they work everywhere. (#52)
- **404 → claim funnel.** A dead `/<username>` is now an invitation: "this patch
  is empty", the name itself, and a "claim this username" call to action, on the
  brand shell. Still returns HTTP 404, still cookie-free and zero-JS. (#52)

## Login

- **Log in with username *or* email.** The login field accepts either; they
  can't collide (a username has no `@`, an email must), so it's a single
  unambiguous lookup and adds no account-enumeration surface. (#54)

## Accessibility

- The auth submit pills carry dark ink, not white-on-accent (2.36:1 fails AA) —
  the same `--accent-ink` token the landing and dashboard already use.

## Notes

- **Theme v3 migration.** The page-settings work bumped the stored theme shape
  to v3 with a migration on load; older themes upgrade transparently. (#55)
- **The design spec lives in-repo** now (`design/`), so the source of truth for
  §B (auth/public states) ships with the code. (#56)
- Deploy is a standard `fly deploy` — no new infrastructure.
- Out of cycle: the magic-link **backend** is deferred to v0.6.0, along with the
  post-launch QA pass (#34).

---

**Build story (raw material — owner supplies voice + facts):** this cycle
finished a refresh that had been split in two so each half stayed reviewable.
The reusable piece was a genuine shared *shell*, not a copy-paste: the auth
pages and the dashboard now pull the same ambient-motif partial, while the
landing keeps its own inline version on purpose — it runs under the strict
zero-JS public CSP and can't load the dashboard stylesheet, so sharing there
would have broken the performance budget. The new state screens were
deliberately built **without** JavaScript: the suggestion chips are `?claim_try=`
links rather than scripted buttons, which kept the dashboard's vanilla-JS file
exactly at its 200-line budget and made the whole funnel work with scripting
off. Two correctness details worth keeping: every suggested username is
re-validated and checked against both live names and resting tombstones before
it's shown (no chip that can't actually be claimed), and the 404 funnel only
offers to claim well-formed, unclaimed names — reserved and malformed paths stay
a plain "nothing here".
